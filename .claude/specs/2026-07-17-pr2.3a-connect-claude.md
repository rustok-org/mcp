## Скоуп PR — что входит / что ЯВНО не входит

**Входит:**
1. `rustok connect claude [--agent <name>] [--force]` — регистрация MCP-сервера у Claude
   Code (scope `user`), preflight-проверки, идемпотентность.
2. RPC-секреты на podman: `rustok-rpc-<agent>-<chain>` (см. решение №2) + их создание/
   обновление из `RUSTOK_RPC_URLS_<chain>` окружения; доставка `--secret …,type=env,
   target=RUSTOK_RPC_URLS_<chain>`. Docker-фолбэк: литеральный `-e` в регистрации +
   печатное tier-примечание (ратифицировано раундом-2 Гейта-1 эпика).
3. Закрытие INTERIM в `start`: `wallet_run_args` перестаёт форвардить по имени те
   `RUSTOK_RPC_URLS_*`, для которых существует RPC-секрет агента (podman), и подключает
   секреты; обещание из комментария `rustok_env_names` исполняется, комментарий и
   `usage()` обновляются.
4. Тесты: стаб `claude` + строгие ассерты (ниже), red-пруфы мутациями.
5. CHANGELOG-запись.

**ЯВНО НЕ входит:** `connect cursor` / `connect hermes` (PR-2.3b — `connect <не-claude>`
получает именованный отказ «coming in PR-2.3b»); `update`/`uninstall` (PR-2.3c);
digest-пин образа (PR-3.2); переписывание INSTALL/README под one-command (PR-4.1 —
существующие доки не трогаю, кроме нуля); правки образа/entrypoint (ноль); `.claude.json`

## Решения — предрешены, к ратификации

1. **Имя MCP-сервера = `rustok`** (ратифицировано эпиком; раунд 2 — переоткрыто НЕ было,
   уточнена реальность). Проба показала: живая запись Капитана УЖЕ называется `rustok` (не
   `rustok-wallet-tui` из `docs/INSTALL.md:72`) — коллизия имён решается штатной
   идемпотентностью №6 (`--force`), а env-file-эпоха ловится отдельно: запись `rustok*`,
   содержащая `--env-file`, → громкая подсказка миграции (см. раздел «Миграция»).
   Doc-эпохальное имя `rustok-wallet-tui` при обнаружении — предупреждение с командой
   удаления, НЕ авто-удаление (чужую запись молча не трогаем).
2. **RPC-секрет — пер-чейн: `rustok-rpc-<agent>-<chain>`** (например
   `rustok-rpc-claude-1`), target — ровно его переменная. Эпик писал `rustok-rpc-<agent>`
   в единственном числе, но сам же целился в `target=RUSTOK_RPC_URLS_*` — переменных
   несколько по построению (`RUSTOK_RPC_URLS_<chain>`, CONFIGURATION.md:12: chain-id в
   имени). Один секрет не может нести N переменных с разными target'ами — пер-чейн имя
   единственная форма, реализующая ратифицированный механизм. **Отклонение от буквы эпика,
   не от решения** — флажу явно.
3. **RPC-секреты обновляются без `--force` — и ТОЛЬКО через `secret create --replace`**
   (раунд 2, ответ на вопрос Ревьюера + его пункт №4). Проба на этой машине (podman
   5.8.4): `secret rm` на занятом секрете успешен МОЛЧА — rm+create имел окно «секрета
   нет», где упавший connect оставил бы регистрацию, ссылающуюся на несуществующий секрет
   (следующая сессия агента не стартует). `--replace` атомарен, окна нет; `rm` в connect
   не используется вообще. Семантика ротации (доказано пробой): живой главный процесс
   держит старое значение до перезапуска, новые контейнеры/exec-процессы получают новое.
   Значение — по-прежнему builtin `printf` в stdin, вне argv. Переменная не задана,
   секрет есть → секрет остаётся и едет в регистрацию (источник истины — секрет-стор).
4. **Не-секретные `RUSTOK_*` (например `RUSTOK_ALLOWED_CHAINS`) — литеральным
   `-e NAME=value` в регистрации.** Name-only `-e` здесь НЕ работает: контейнер запускает
   **Claude Code, из СВОЕГО окружения** (не шелл человека) — наследовать нечего; в
   `rustok start` name-only корректен, потому что там запускает шелл человека (раунд 2:
   фраза была перевёрнута — исправлена). **Заморозка (раунд 2):** литеральные значения —
   снимок окружения НА МОМЕНТ connect; смена значения = повторный `connect --force`
   (печатается в выводе connect). Единая семантика без спец-кейсов: замораживается ВСЁ
   `RUSTOK_*` из окружения, кроме `RUSTOK_KEYRING_PASSWORD(_FILE)` (только секрет/файл),
   `RUSTOK_IMAGE` (ручка шима) и `RUSTOK_RPC_URLS_*` (едут секретами, №2/№3) — включая
   `RUSTOK_MCP_CAPABILITIES`, если задана (конфликт с «Замечено» снят: там осталась лишь
   строка «не изобретаю значение по умолчанию»).
5. **`connect` требует `init` заранее** (secret+volume существуют — те же пробы, что у
   `start`, `cli/rustok:298-300`). Отказы дифференцированы (раунд 2): volume есть,
   секрета нет → это env-file-эпоха, отказ с миграционной подсказкой (`rustok init
   --force` → `connect --force`, см. «Миграция»); нет ни того ни другого → `rustok init`.
   **Warn о множественности перестроен на реальный источник (раунд 2, пункт №3):**
   множественность порождают агент-сессии (резидентная модель: N сессий = N контейнеров),
   а не `rustok start`, и живые контейнеры БЕЗ `rustok.agent` — discovery по двум меткам
   их не видит. Поэтому проба — по настоящему домену коллизии, ТОМУ:
   `ps --filter volume=<volume_for(agent)>` (фильтр проверен пробой на 5.8.4). N>0 →
   предупреждение ФАКТОМ, без ложной уверенности: «N контейнер(ов) уже используют кейстор
   '<vol>' (каждая агент-сессия поднимает свой); одновременная работа нескольких на одном
   кейсторе не проверена — закрой лишние сессии». Класс сам по себе — эскалация №1,
   решается вне PR.
6. **Идемпотентность `connect`:** сервер `rustok` уже зарегистрирован (проба —
   `jq -e '.mcpServers.rustok' ~/.claude.json`, **read-only по файлу конфига**) →
   именованный отказ; `--force` = `claude mcp remove -s user rustok` + `add` заново.
   **Раунд 3 — почему файл, а не `claude mcp get`:** у `claude mcp` НЕТ не-health-check'ающей
   команды чтения — и `get`, и `list` подключаются к серверу, т.е. поднимают кошелёк-контейнер
   на общем кейсторе за один вопрос «я зарегистрирован?» (непроверенный класс, эскалация №1).
   Честная цена (проговорена Ревьюером, принята): `~/.claude.json` — внутренний формат
   поддерживаемого интерфейса, завязка на него; запись — по-прежнему ТОЛЬКО через
   `claude mcp add/remove` (там побочка и есть цель). `connect` требует `jq` — preflight
   с именованным отказом + подсказка в `doctor`; остальные команды шима без jq как были.
   Для протокола, добыто пробой (fake-HOME, спорный контейнер не поднимался): коды `get` —
   нет в конфиге → 1; есть-но-сломан (`/bin/false`) → **0** (`✘ Failed to connect` в тексте);
   подключился → 0. Т.е. exit-код `get` различает существование, не здоровье — опасение
   «add поверх сломанной записи» через код возврата не материализуется; уходим по главной
   причине (контейнер), вопрос закрыт фактом, не рассуждением.
   **Полный контракт jq-пробы (раунд 4, таблица Ревьюера, реплицирована моей пробой):**
   `0` — запись есть → занято; `1` — записи нет (включая пустой `{}`) → свободно;
   `2` — файла нет вообще → **свободно** (свежая машина — главная цель эпика; отказ здесь
   отшил бы именно нового пользователя); `5` — битый JSON → **именованный отказ** (молча
   счесть «свободно» нельзя — `mcp add` поверх сломанного конфига; чинит человек, не шим).
   Реализация: `0→занято; 1|2→свободно; *→отказ` (любой прочий код — как 5: не понял файл,
   не трогай). **Скоуп пробы — намеренно только user-scope** (`~/.claude.json`): ровно то,
   чем шим управляет (`-s user`) и принцип «чужую запись молча не трогаем»; `--mcp-config`/
   project-scope/managed-policy проба не видит — `get` платил контейнером в т.ч. за
   кросс-скоупный обзор, мы от него отказались осознанно (НИТ Ревьюера, зафиксировано). Симметрия с `init --force`
   (пересоздаётся только регистрация, как там — только секрет). **Порядок операций
   (раунд 2, следствие пробы №3):** сначала ВСЕ секреты (`--replace`, идемпотентно),
   потом регистрация (remove→add) — упавший между секретами connect оставляет старую
   рабочую регистрацию нетронутой. Если `add` упал после `remove` — печатается точная
   команда `claude mcp add …` для ручного повтора (все аргументы уже вычислены);
   молчаливого полу-состояния нет.

## Миграция живого пользователя (Капитан) — ЧАСТЬ ЭТОГО PR (раунд 2, пункт №1)

Чей это PR: **этот** — connect без миграционного пути неисполним на единственной живой
машине (Блокер 1). Код: дифференцированный отказ №5 (volume есть + секрета нет → печать
шагов миграции). Runbook (исполняет Капитан, я веду; предусловие — решение эскалации №1):

0. **Точка отката:** `jq '.mcpServers.rustok' ~/.claude.json > ~/rustok-mcp-backup-<дата>.json`
   (вне репо) — точный JSON, восстановимый механически (проверено на живом конфиге: 11
   аргументов извлекаются как есть). Read-only: контейнер НЕ поднимается, шаг 0 честно
   стоит ДО «закрыть сессии» (раунд 3: `claude mcp get` здесь был тройной ошибкой — он
   health-check'ает как `list`, поднял бы кошелёк до шага 1 и отдал бы прозу вместо JSON;
   из runbook'а исключён полностью). Откат в любой момент =
   `claude mcp remove -s user rustok` + `add` из сохранённого JSON.
1. **Закрыть агент-сессии** → `podman ps --filter volume=rustok-wallet-tui` до нуля
   (`-i --rm`-контейнеры умирают с сессией; это же закрывает мульти-writer окно на время
   миграции).
2. `rustok init --force` — Капитан вводит СВОЙ существующий пароль кейринга на tty →
   секрет `rustok-keyring-claude` создан; **кейстор-том не затрагивается** (гарантия
   `init --force`, тест 30/40 сюиты).
3. Экспортировать RPC-переменные в шелл (значения у Капитана; шим legacy-файл НЕ парсит —
   решено против авто-импорта: меньше поверхности, файл уходит в утиль эскалацией №2) →
   `rustok connect claude --force` — секреты `rustok-rpc-claude-{1,8453}` созданы,
   регистрация заменена (порядок №6: секреты до регистрации).
4. Verify — см. DoD.
5. Судьба `wallet.env`/`wallet.envy` — **эскалация №2, решение Капитана** (рекомендация
   Ревьюера: `envy` снять сразу, `env` — после подтверждённой миграции). Шим их не трогает.

## Критерии приёмки — «PR готов, когда…»

- `rustok connect claude` на podman-машине с пройденным `init` регистрирует сервер одной
  командой; собранный argv **дословно** совпадает с эталоном (см. тест-план №1).
- Ключ провайдера RPC отсутствует и в argv `claude`, и в `~/.claude.json` (podman-путь).
- `rustok start` на podman с существующим RPC-секретом больше не форвардит эту переменную
  по имени (INTERIM закрыт), контейнер получает значение через secret.
- Docker-путь: регистрация с литеральным `-e`, tier-примечание печатается.
- Все отказы именованные: не тот клиент (`connect cursor` → «PR-2.3b»), нет init, повторный
  connect без `--force`, невалидный `--agent` (существующий charset-гейт покрывает connect).
- Сюита шима зелёная целиком (старые 46 + новые), shellcheck (пин v0.9.0) чист,
  `uv run ruff/mypy/pytest` зелёные (питон не трогается — но гейты гоняются все, полный
  набор `ci.yml`).

## Тест-план (строгие ассерты, прецедент `assert_resident`)

1. **Точный argv регистрации (podman):** стаб `claude` пишет argv в лог; ассерт —
   **дословное** сравнение всей строки `mcp add -s user rustok -- podman run -i --rm
   --init --label rustok=wallet --label rustok.agent=claude -v rustok-wallet-tui:/data
   --secret rustok-keyring-claude,type=env,target=RUSTOK_KEYRING_PASSWORD --secret
   rustok-rpc-claude-1,type=env,target=RUSTOK_RPC_URLS_1 -e RUSTOK_ALLOWED_CHAINS=1
   ghcr.io/rustok-org/rustok-wallet-tui:v0.7.1` (полное равенство, не grep-подстроки —
   всеядность уже дважды пропускала баги).
2. **Секрет-механика:** значение `RUSTOK_RPC_URLS_1` с кавычками/`$`/`&` доехало в
   стаб-стор байт-в-байт; значение ∉ `calls.log` (вне argv); ассерт «RPC-значение ∉ argv
   `claude`» отдельной строкой.
3. **Негативные контроли:** ничего нет → отказ «rustok init»; volume есть + секрета нет
   → отказ с миграционной подсказкой (`init --force`) — отдельный тест, это Блокер-1-путь;
   повторный connect без `--force` → именованный отказ (тест сажает запись `rustok` в
   `$WORK/home/.claude.json` — проба шима читает ФАЙЛ, стаб `claude` не спрашивается);
   отсутствие jq в PATH → именованный отказ с подсказкой (только connect);
   `$HOME/.claude.json` ОТСУТСТВУЕТ (свежая машина, jq exit 2) → connect ПРОХОДИТ (это
   главный пользователь эпика); битый JSON (jq exit 5) → именованный отказ, `mcp add` в
   логе стаба ОТСУТСТВУЕТ (раунд 4);
   `--force` → порядок в логе: secret --replace… ПОТОМ mcp remove, mcp add (№6);
   `connect cursor` → «PR-2.3b»; контейнер на томе агента (стаб-таблица, БЕЗ
   `rustok.agent`-метки — как живые) → warn-строка с числом и именем тома; doc-эпохальное
   имя в стаб-конфиге → warn с командой удаления. Docker-путь: литеральный `-e` в argv,
   tier-строка напечатана.
4. **`start`+секреты:** podman-стаб с существующим `rustok-rpc-claude-1` → argv `start`
   содержит `--secret rustok-rpc-claude-1,…` и НЕ содержит `-e RUSTOK_RPC_URLS_1`;
   без секрета — прежнее name-only поведение (обратная совместимость до Стадии 4).
5. **Red-пруфы (мутации, по одному на инвариант):** m1 — RPC-значение в argv вместо stdin
   → тест 2 красный; m2 — потеря второй метки в регистрации → тест 1 красный; m3 —
   авто-`mcp remove` без `--force` → тест 3 красный; m4 — name-only `-e` вместо литерала
   в регистрации → тест 1 красный; m5 (раунд 2) — `rm+create` вместо `--replace` → тест
   порядка красный (стаб-лог содержит `secret rm` — запрещён в connect целиком). Для
   каждой мутации прикладываю diff мутанта + красный вывод (урок: докажи, что проба
   ПАДАЕТ).
6. **Живой `claude mcp list → Connected`** — НЕ в сюите: трогает реальный конфиг и живой
   кошелёк Капитана. Исполняется как деплой-verify (DoD) по слову Капитана. В сюите — вся
   механика до границы `claude`-бинаря; это честно флажу как «Connected проверяется живьём,
   не стабом».
