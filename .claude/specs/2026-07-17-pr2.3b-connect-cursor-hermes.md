## Цель — один абзац

Довести «одну команду» до двух оставшихся клиентов. `connect cursor` пишет запись в
`~/.cursor/mcp.json` (у Cursor нет CLI-регистратора — формат подтверждён доками:
`mcpServers` → `{command, args}`, тот же, что у Claude). `connect hermes` оформляет
Стадию 0 в команду: свой кошелёк Hermes (том `rustok-hermes`, суб-метка), запись в
`mcp_servers` его `config.yaml`, `enabled: true` — и хоронит для внешних пользователей
Стадии-0-эпохи два бага той эпохи: запись `rustok-wallet` с
`args`-JSON-строкой-вместо-списка и обёртку `scripts/rustok-mcp-server.py` (на ЭТОЙ
машине оба уже вычищены сносом 17.07 — см. баннер раунда 2; код и тесты держат путь
для тех, у кого состояние живо).

## Скоуп PR — что входит / что ЯВНО не входит

**Входит:** `connect cursor` + `connect hermes` (оба: `[--agent] [--force]`);
рефактор `cmd_connect` в «общий преflight+argv → пер-клиентный writer»; тесты обоих
путей с побайтовыми ассертами и мутациями по прод-строкам; CHANGELOG.

**ЯВНО НЕ входит:** `update`/`uninstall` (2.3c) · инсталлер (3.2) · переписывание доков
(4.1) · миграция живого claude-кошелька (runbook 2.3a, ждёт эскалаций) · любые касания
демона Hermes (только печатная инструкция «перезапусти») · удаление файлов Hermes
(обёртка — только hint) · Windows/проектные (`.cursor/` в репо) скоупы Cursor.

## Решения — предрешены, к ратификации

1. **Дефолтный кошелёк = имя клиента**: `connect cursor` → `--agent cursor` (том
   `rustok-cursor`), `connect hermes` → `--agent hermes` (том `rustok-hermes`).
   Симметрия с 2.3a (`connect claude`→claude) и решение эпика №4 «каждому агенту свой
   кошелёк» (общий том = гонка nonce, непроверенный мульти-writer). Явный `--agent
   claude` разрешён (сознательная кросс-проводка) — volume-warn из 2.3a её осветит.
   Следствие: перед `connect cursor` нужен `rustok init --agent cursor` (новый сид).
2. **Cursor: шим пишет `~/.cursor/mcp.json` сам, через jq** — CLI-регистратора у Cursor
   нет, это НЕ отход от «пиши через CLI» 2.3a, а его отсутствие у клиента. Контракт
   пробы тот же (`jq -e`: 0 занято / 1|2 свободно / * именованный отказ); файла нет →
   создаётся с `{}`; запись через `jq '$ARGS.positional' --args` **строго с
   терминатором `--`** (блокер 3, подтверждён пробой: без `--` jq падает на первом же
   флаге argv — `Unknown option -i`; с `--` всё едет позиционно, включая значения с
   пробелами/эскейпами) — argv без línea-канала; атомарно tmp+mv;
   старая запись `rustok` печатается как путь возврата ДО замены (паттерн old_entry
   2.3a). Имя записи — `rustok`. Чужие записи в файле не трогаются байт-в-байт
   (jq перепишет форматирование ТОЛЬКО затронутого файла — cursor-файл целиком наш
   формат; чужие КЛЮЧИ сохраняются значениями как были).
3. **Hermes: простой YAML round-trip** (раунд 2 — хирургия по маркам снята вместе с
   требованием комментариев, решение Капитана): `python3` + PyYAML,
   `safe_load → правка mcp_servers → safe_dump(sort_keys=False)`, бэкап
   `config.yaml.rustok-bak-<ts>` до записи, атомарно tmp+mv; неразборный YAML →
   именованный отказ («не понял файл — не трогай»). Механизм проверен вживую сносом:
   Ревьюер именно так удалил `rustok-wallet` — 22 рабочих ключа уцелели, файл валиден.
   Требования: `python3`+PyYAML — именованный отказ без них (у живого Hermes PyYAML
   есть по построению; проба: yaml 6.0.3 на машине). `mcp_servers` отсутствует →
   создаётся.
4. **Наша legacy-запись `rustok-wallet` у Hermes удаляется в той же записи блока** +
   печатается note и hint снести обёртку (`rm ~/.hermes/scripts/rustok-mcp-server.py`
   — сам файл НЕ трогаем). Почему удаление, а не warn, как doc-era у claude: запись
   создана НАШИМ runbook'ом (Стадия 0), она мертва (`enabled: false`, args-баг) и живёт
   в блоке, который мы всё равно переписываем; оставить = два rustok-входа у одного
   клиента. Это НЕ «чужая запись» — чужие ключи `mcp_servers` (если появятся)
   сохраняются как были.
5. **Форма записи**: cursor `{"command": "<engine>", "args": [...]}`; hermes
   `rustok: {command: <engine>, args: [...], enabled: true}` — argv ОДИН на всех
   клиентов, собирается общим кодом 2.3a (label_volume_args / password_args_for /
   rpc_secret_args / инлайн `-e`-цикл), RPC — пер-агентными секретами своего агента
   (`rustok-rpc-cursor-*` и т.д.), пароль — секретом своего агента. `env`-ключ и
   env-file в записи не появляются вовсе (hermes.env из старого runbook'а НЕ нужен).
6. **Структура кода**: `cmd_connect` = общий преflight (движок, init-пробы,
   мультипликативный volume-warn, секреты `--replace`) + сборка argv «podman run …»
   один раз в `"$@"` + диспетч `connect_write_<client> "$@"` (позиционные параметры —
   единственный POSIX-канал, переживающий любые значения). Пер-клиентные пробы:
   jq-контракт для claude/cursor (разные файлы), python-yaml-проба для hermes.
   Инструменты: jq нужен claude+cursor-путям, python3+yaml — только hermes-пути;
   `doctor` дополняется строкой про python3+yaml (info, не fail).
7. **Рестарт клиента — печатная инструкция**, демоны не трогаем: hermes живой
   (`gateway.pid`), убийство процессов агента — не дело шима (прецедент: health-check
   убил бы живой кошелёк).

## Критерии приёмки

- `connect cursor` на подготовленном (`init --agent cursor`) стенде создаёт/дополняет
  `~/.cursor/mcp.json`; запись `rustok` **структурно равна эталону** (`jq -cS` compare);
  чужие ключи файла сохранены; файла не было → создан.
- `connect hermes`: блок `mcp_servers` заменён (наша запись `rustok`, `enabled: true`,
  args НАСТОЯЩИМ списком), `rustok-wallet` исчез с печатной note, **все top-level ключи
  вне `mcp_servers` пережили round-trip со своими значениями** (МИНОР-1 раунда 2:
  клауза о комментариях снята — механизм №3 их не сохраняет по ратифицированному
  решению), бэкап создан; hint обёртки напечатан.
- Все отказы именованные: нет init (+ env-file-эра для cursor/hermes аналогично 2.3a),
  занято без `--force`, битый JSON/YAML, нет jq / нет python3+yaml, неизвестный target.
- Сюита целиком зелёная; мутации — по прод-строкам обоих writer'ов (см. тест-план);
  shellcheck (пин 0.9.0) чист; ruff/mypy/pytest не тронуты, но гоняются все.

## Тест-план (мутации — обходом прод-строк, урок раунда 2 Гейта-2)

Классы: (а) позитив cursor — точный `jq -cS`-эталон записи + сохранение чужого ключа +
создание файла из ничего + бэкап-вывод старой записи при `--force`; (б) позитив hermes —
эталон блока (python-compare в тесте), **round-trip рабочих ключей: все top-level ключи
вне `mcp_servers` переживают запись со своими значениями** (раунд 2 — вместо
комментарий-стража), `rustok-wallet` удалён + note, бэкап-файл существует, args —
НАСТОЯЩИЙ список (регресс-тест бага Стадии 0), `enabled: true`; (в) негативы — по
одному на каждый именованный отказ обоих путей (вкл. «нет python3», «нет yaml», битый
YAML, битый JSON, занято-без-force); (г) **мутационная карта: для КАЖДОЙ новой
прод-строки-гварда — своя мутация с red-пруфом**, минимум: **потеря `--` после
`--args`** (страж: round-trip argv с ведущими дефисами — `run -i --rm …` доезжает до
JSON целиком; блокер-3-тест), подмена атомарного mv на прямую запись, недефолтный
agent игнорируется (страж: эталон с `--agent cursor`→том), `enabled: true` потерян,
legacy-запись оставлена, round-trip теряет чужой top-level ключ. Плюс перегон прежней
батареи 2.3a поверх рефактора диспетча. Red-фаза: все новые тесты красные до
реализации (диспетча не существует).
