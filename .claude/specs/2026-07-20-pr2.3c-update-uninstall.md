## Цель — один абзац

Замкнуть жизненный цикл «одной команды» — последние два глагола Стадии 2. `update`
подтягивает свежий образ кошелька и перерегистрирует НАШИ MCP-записи у всех
подключённых клиентов, сохраняя каждому его выбранный кошелёк; сам шим себя не
обновляет — self-update = повторный прогон инсталлера (печатной строкой, механизм
приедет с 3.2). `uninstall` — data-safe демонтаж в обратном порядке установки:
регистрации, контейнеры, секреты, PATH-блок, сам шим — но **кейстор-тома (= ключи =
деньги) не трогаются никогда**, кроме отдельного пути с двойным гейтом (флаг +
интерактивное подтверждение литералом с /dev/tty).

## Скоуп PR — что входит / что ЯВНО не входит

**Входит:** `cmd_update` + `cmd_uninstall` (включая гейтованный purge-путь); **фикс
hermes-писателя: печать старой записи при `--force`-замене (дельта Гейта-1)**; эволюция
статического инварианта томов (ровно один call-site); расширение стаба (`volume rm`/`ls`);
тесты с мутационной картой; `usage()`; CHANGELOG; **PATH-маркер как контракт для 3.2**.

**ЯВНО НЕ входит:** `install.sh` (3.2; здесь только определение маркера); digest-пин
(3.1/3.2 — update работает с `image_ref()` как есть: тег или RUSTOK_IMAGE-override);
переписывание доков (4.1 — здесь только usage/CHANGELOG); self-update шима кодом;
Windows/macOS; core-код (ноль).

## Решения — предрешены, к ратификации

1. **`update` = pull → перерегистрация всех НАШИХ записей с сохранением агента.**
   Порядок жёсткий: `$engine pull "$(image_ref)"` — отказ здесь = СТОП до касания
   конфигов; затем для каждого из трёх клиентов: есть наша запись `rustok` → извлечь
   `rustok.agent=<name>` из args САМОЙ записи (одно место правды) → перегнать
   писатель connect с `--agent <name> --force`; клиент не подключён → пропуск
   (в конце «updated N client(s)»). Частичный провал: остальных доделываем,
   именованный свод, exit 1.
   Самоатака: перерегистрация перетирает ручные правки пользователя в нашей записи —
   не перевесило: запись наша по контракту 2.3a («шим управляет ровно тем, что пишет»),
   `--force`-семантика ратифицирована. Цена ошибки: идемпотентная перезапись той же
   записи. Факт-переключатель: клиент, у которого перезапись рвёт живую сессию агента.
   **Дельта Гейта-1 (находка Ревьюера, вшита):** update — рутинная команда, тихое
   поглощение ручной правки недопустимо → **старая запись печатается перед заменой у
   КАЖДОГО тронутого клиента** (recovery-путь, единый у всех трёх писателей). Для этого
   **hermes-писатель чинится в этом же PR**: при `--force`-замене существующей записи
   `rustok` печатает старый блок (python уже парсит конфиг — извлечь блок до правки;
   файл-бэкап остаётся как был). **Уточнение реализации (симметрия, тот же принцип):
   claude-писатель сегодня печатает old_entry только при ПРОВАЛЕ `mcp add` (:614–617) —
   успешная замена молчит; чинится тем же PR: печать перед remove→add, как у cursor
   (:636–637), печать-при-провале остаётся.
2. **Запущенный кошелёк НЕ перезапускается** — печатное уведомление «wallet running on
   the old image — new image applies on the agent's next start (or: rustok stop)».
   Контейнер эфемерный; тихий рестарт под ногами агента = потерянная сессия.
3. **self-update шима = re-run инсталлера, НЕ код** (ратифицировано эпиком, НИТ р.1).
   До 3.2 — печатная строка-hint в выводе update.
4. **`uninstall` дерегистрирует у клиентов — РАСШИРЕНИЕ строки эпика, флажу.**
   Эпик: «secret rm + PATH + том-гейт». Рекомендация: включить дерегистрацию.
   Почему: uninstall, оставляющий записи на удалённые секреты, ломает следующий старт
   каждого агента громкой ошибкой контейнера — «удалил rustok, сломались агенты».
   Дёшево: писатели уже есть (`claude mcp remove` / jq del + tmp+mv / YAML round-trip).
   Самоатака: «минимум скоупа» — но цель uninstall = система как до нас, а регистрации
   писали мы. Цена, если не прав: ~30 лишних строк с готовыми тестами.
   Факт-переключатель: слово Капитана «строго по строке эпика».
5. **Том — двойной гейт (эпик: «только по второму подтверждению»).** `uninstall` тома
   не трогает и печатает: «keys intact: volumes X, Y (remove later:
   rustok uninstall --purge-keys)». Снос = `uninstall --purge-keys`: (а) флаг,
   (б) интерактив с /dev/tty — печать списка томов + требование ввести литерал
   `delete my keys`; нет tty → именованный отказ (Rule of two windows, как у init).
   Никаких `-y`-обходов. Формулировка prompt'а точная (заметка Ревьюера): «deleting
   these volumes destroys the keys; without your seed phrase backup the funds are
   unrecoverable» — «невосстановимо» привязано к СНОСУ, не к самому тому (цел том +
   пароль keyring = второй путь восстановления через `init --force`).
   **Честный контракт гейта (security-pass BLOCKER, вшит):** tty-гейт останавливает
   пайп, слепую `-y`-автоматизацию и агента, дёргающего rustok как инструмент — но НЕ
   процесс, аллоцирующий свой pty (`script(1)`, `pty.spawn`). Абсолютной гарантии нет:
   у кого есть shell — дойдёт до `podman volume rm` без шима. Комментарий в коде и
   CHANGELOG сформулированы честно (не «агент никогда не сможет»); гейт стережёт дорогу
   ШИМА, движок — доверенная граница платформы.
6. **Статический инвариант томов эволюционирует, не умирает.** Тест «shim code never
   removes volumes» → «`volume rm` встречается РОВНО один раз — внутри purge-блока
   cmd_uninstall» (grep-count по исходнику без комментариев + контекст-ассерт) +
   динамические негативы (без флага/без подтверждения тома целы). `--name`-инвариант
   не трогается (0 вхождений).
7. **Порядок демонтажа без гейта (не-money):** дерегистрация → стоп контейнеров
   `rustok=wallet` (все агенты, перечислить) → секреты нашего префикса
   (`secret ls` → `rustok-keyring-*`, `rustok-rpc-*`; docker: файлы
   `$CONFIG_DIR/keyring-pass-*`) → PATH-блок по маркеру → сам шим
   (`~/.local/bin/rustok`) + `$CONFIG_DIR` последними (не потерять engine-pin посреди
   работы). Динамическая самопроверка после каждого шага, свод в конце.
8. **PATH-маркер определяется СЕЙЧАС как контракт для 3.2:** инсталлер обязан писать
   блок между `# >>> rustok installer >>>` и `# <<< rustok installer <<<`; uninstall
   удаляет ровно блок между маркерами в стандартных профилях (`.bashrc`/`.zshrc`/
   `.profile`), маркеров нет → печатный hint, чужие rc-файлы вслепую не редактируем.
   **Дубль-маркер (security-pass MINOR, вшит):** ровно ОДНА пара маркеров или отказ —
   `sed`-диапазон не пере-взводится на втором `>>>`, слепое удаление съело бы строки
   пользователя между двумя стартами; при `starts≠1||ends≠1` — именованный warn, файл
   не тронут (тест-страж + red-мутация).

## Критерии приёмки — «PR готов, когда…»

- **update:** pull вызван с `image_ref()` ДО записей; каждая существующая запись
  перерегистрирована со СВОИМ агентом (byte-exact эталоны против писателей #74/#75);
  **старая запись напечатана по каждому тронутому клиенту (все три писателя, включая
  починенный hermes)**; неподключённые клиенты пропущены; pull-fail → конфиги нетронуты
  байт-в-байт, exit 1; частичный фейл → свод + exit 1; оба уведомления (running-wallet,
  self-update hint) печатаются; `нет ни одной записи → «nothing to update», exit 0`.
- **uninstall:** после прогона регистраций нет (чужие ключи конфигов целы —
  round-trip-стражи #75 реюзаются), контейнеры остановлены, секретов префикса нет
  (podman и docker-путь), PATH-блок снят ровно по маркерам (соседи целы), шим+config
  удалены; **тома ЦЕЛЫ** + печать «keys intact».
- **purge-путь:** без tty → именованный отказ; неверный литерал → отказ, тома целы;
  верный литерал → удалены ровно НАШИ тома, печать итога.
- **Статика:** `volume rm` = ровно 1 call-site в purge-блоке; `--name` = 0.
- Каждая новая прод-строка-гвард — мутация с red-пруфом; сюита целиком зелёная;
  shellcheck (стаб + шим + сюита) чист; ruff/mypy/pytest не тронуты, но гоняются все.

## Тест-план (мутации — обходом прод-строк, дисциплина raунда 2 Гейта-1 2.3b)

(а) **update-позитив:** fixtures всех трёх клиентов → после update запись каждого с
тем же агентом и свежим argv (byte-exact); **старая запись напечатана по каждому из
трёх** (маркер-значение в fixture — ассерт по stderr); override-fixture (cursor-клиент
с `rustok.agent=claude`) сохраняет claude — реюз стража #75 (уточнение Ревьюера: это
страж пути ЧТЕНИЯ агента update'ом, он не подменяет закрытый МИНОР #75 на путь
ЗАПИСИ `connect --agent`). Мутации: потеря `--force` у перерегистрации · потеря
извлечения агента (запись съехала на дефолт → red по эталону) · перестановка pull
ПОСЛЕ записей (pull-fail, конфиг тронут → red byte-compare) · **потеря печати старой
записи (red по stderr-ассерту)**.
(б) **update-негативы:** pull-fail → exit 1 + конфиги байт-в-байт; ноль записей →
«nothing to update» exit 0.
(в) **uninstall-позитив:** полный демонтаж, тома целы, печать; docker-путь — файлы
паролей сняты. **Плюс hermes-писатель напрямую: `connect hermes --force` над
существующей записью печатает старый блок (новый тест, зеркало claude/cursor-ассертов
#74/#75; мутация: потеря печати → red).**
(г) **uninstall-негативы:** без `--purge-keys` тома целы (динамика поверх статики);
без tty → отказ; неверный литерал → отказ, тома целы. Мутации: пропуск tty-гейта ·
подмена литерала · `volume rm` вне purge-блока (статика red).
(д) **purge-позитив:** верный литерал → `volume rm` ровно для наших томов.
(е) **PATH:** fixture-профиль: блок между маркерами удалён, соседние строки целы;
без маркеров — hint, файл нетронут.
Red-фаза: все новые тесты красные до реализации (глаголов не существует).
