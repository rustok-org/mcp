# Этап 6 — E2E pty-приёмка канала одобрения (репо mcp)

> Родительская спека: `core/.claude/specs/2026-07-05-slice1b-tui-approver.md` §Этап 6 (стр. 275),
> §7 (гейты), §8 (риски). Эта спека — исполнение §Этапа 6 **с поправкой на отгруженную
> реальность**: «поезд» (console v0.1.0 · core v0.2.0 · wallet-tui 0.5.0 → 0.6.0) уже уехал,
> сценарии читаются против образа `ghcr.io/rustok-org/rustok-wallet-tui:v0.6.0` (digest `d75e5fe`).

## Цель

Доказать pty-harness'ом, что канал одобрения работает **на том артефакте, который отгружен
пользователю**, а не только в юнит-тестах трёх репозиториев. Этап приёмочный: прод-код не
меняется ни в mcp, ни в core, ни в console. Продукт этапа — воспроизводимая E2E-сюита в репо
`mcp` + полный лог прогона в отчёт (требование §7 родительской спеки) + upgrade-доки.

## Скоуп PR

**Входит:**
- E2E-сюита `tests/e2e/` (pytest, маркер `e2e`) против образа `rustok-wallet-tui:v0.6.0`:
  7 сценариев родительской спеки + S0 (podman-tmpfs `/run/wallet`).
- Фикстуры: локальная цепь (anvil в контейнере), одноразовый том/кошелёк, podman-сеть,
  MCP-stdio-клиент (JSON-RPC по пайпам), pty-драйвер консоли (pexpect + pyte).
- Dev-зависимости `pexpect`, `pyte` в `pyproject.toml` (группа dev) + `uv.lock`; там же — маркеры,
  `addopts` и `per-file-ignores` (Р-E).
- `docs/` — секция «Upgrading the wallet image» (Р-G).
- `CHANGELOG.md` — секция `## [Unreleased]` (одна строка про upgrade-доки). Рантайм не бампается:
  версии-без-релиза в этом файле раньше не было, а PR не меняет прод-код.

**ЯВНО не входит:**
- Любая правка прод-кода mcp/core/console. Если E2E найдёт дефект — он идёт в отчёт как находка;
  фикс — отдельная задача с решением Капитана (иначе мы тестируем не то, что отгрузили).
- Ручка TTL в core (см. Р-C), релизный поезд §Этапа 6 (уехал), листинги tui, демон+compose.
- Запуск E2E в CI (§8: «CI не гоняет интерактивный TUI»; сюита ручная/локальная, маркер `e2e`
  исключён из дефолтного прогона). Подъём в CI — отдельная задача, когда будет podman-раннер.
- Хвосты: `docs/approval-channel-draft`, CI-assert IMAGE-дефолтов.

## Решения к ратификации (Гейт-1)

### Р-A. Цепь для исполняемых сценариев = локальный **anvil** в контейнере
`ghcr.io/foundry-rs/foundry:latest`, `anvil --host 0.0.0.0 --chain-id 31337`; кошелёк
конфигурируется `RUSTOK_ALLOWED_CHAINS=31337` + `RUSTOK_RPC_URLS_31337=http://anvil:8545`;
баланс наливается cheat-кодом `anvil_setBalance`.

- **Почему:** сценарий 1 требует настоящего broadcast и настоящего `tx_hash` — anvil даёт
  ровно это (тот же путь подписи, тот же `eth_sendRawTransaction`), но детерминированно,
  мгновенно, без сети, крана и чужого аптайма. Произвольный `chain_id` — штатная конфигурация
  ядра, не хак: `RUSTOK_ALLOWED_CHAINS` парсится как свободный список u64 (`config.rs:112`),
  `build_providers` поднимает провайдер для любой цепи, у которой есть RPC-URL
  (`providers.rs:25-38`); вайтлиста сетей нет.
- **Контраргумент против себя:** anvil — не настоящая сеть (нет мемпула, газового рынка,
  реорга). Не перевесил: приёмка проверяет **наш** контур (парковка → человек → подпись →
  broadcast → статус), а не поведение чужого мемпула; сетевой риск ловится не приёмочным
  гейтом, а прод-эксплуатацией.
- **Цена ошибки:** если Капитан хочет именно Sepolia — E2E становится флейки и
  неповторяемым (кран, задержки блоков), а гейт, который иногда красный без вины кода,
  быстро научатся игнорировать.
- **Что изменило бы выбор:** требование доказать поведение против конкретного публичного
  RPC-провайдера (тогда — отдельный «сетевой» прогон поверх anvil-сюиты, не вместо неё).
- **Статус: ПРОВЕРЕНО ЗОНДОМ** (см. §Доказательства) — не гипотеза.

### Р-B. Harness = Python `pexpect` + `pyte`, НЕ `expect`/`script`
- **Почему:** `expect` (Tcl), `script` и `socat` на машине отсутствуют; тащить системный пакет
  в Python-репозиторий незачем. `pexpect` даёт pty, `pyte` — эмулятор терминала. **`pyte`
  обязателен, а не украшение:** консоль — полноэкранный TUI, сырой поток pty рвёт слова
  escape-последовательностями (проверено — регекс по сырому потоку не матчится); ассерты
  обязаны идти по **отрендеренному экрану**.
- Обе зависимости ставятся через uv (проверено), harness остаётся в тулчейне репо и позже
  поднимается в CI без системных пакетов.
- **Синхронизация — по отрендеренному состоянию, не по `sleep`.** Консоль поллит `list` раз в
  2.5 с (`main.rs:36`): зонд поймал на этом ложный красный — Enter нажимался, пока на экране
  ещё «Pending approvals: 0». Каждый шаг harness'а ждёт своего экрана с таймаутом.
- **Размер pty ≥ 140×50.** Консоль отказывает в approve, если карточка не влезла
  (`ui.rs:356 priority_fields_fit` — «yes» к нечитаемой карточке не считается решением).
  Дефолтные 80×24 pexpect'а дали бы ложный красный.

### Р-C. Сценарий 3 (истечение) = отдельная «медленная» дорожка, 15 минут реального ожидания
- **Почему:** `DEFAULT_PENDING_TTL` = 15 мин (`approval.rs:29`), ручки в окружении нет
  (`config.rs` знает только DATA_DIR/CHAINS/RPC_URLS/KEYRING_PASSWORD/ALCHEMY/INFURA/VAULTS),
  TTL приходит из `with_default_ttls()` (`server.rs:112`), `force_expire` — `pub(crate)`.
  Ускорить, не меняя core, нельзя.
- **Контраргумент против себя:** можно добавить в core env-ручку TTL. Не перевесил: это новый
  образ — и мы тестируем уже не то, что отгрузили, то есть теряем весь смысл этапа (плюс
  правка прод-кода ради скорости теста — ровно правило №10).
- **Цена ошибки:** 15 минут разового ожидания. Альтернатива (выбросить сценарий) дороже:
  `expired` — состояние, на которое опирается контракт поллинга MCP, и единственное
  E2E-доказательство, что дедлайн реально срабатывает.
- **Деталь harness'а:** подметание ленивое (`sweep` при обращении), фонового таймера нет —
  после ожидания нужен запрос статуса, чтобы `pending` схлопнулся в `expired`.
- **Форма ассерта (МИНОР Гейта-1, проверено по коду):** после истечения запись становится
  `Entry::Resolved`, а её `view()` отдаёт `not_after_unix: 0` (`approval.rs:115-118`); шлюз
  сериализует ноль как `null` (`lib.rs:444`, есть собственный тест на :950). Значит ассерт
  «дедлайн в прошлом» на истёкшем статусе дал бы **ложный красный на 16-й минуте**. Правильно:
  снять `not_after_unix` **при парковке** (состояние ещё `pending`, число живое), убедиться, что
  он в будущем, а после ожидания ассертить `state == "expired"` **и** `not_after_unix is None`,
  сверив, что снятый при парковке дедлайн к этому моменту уже прошёл. Это доказывает, что
  сработал именно дедлайн, а не что-нибудь ещё.
- Маркер `e2e_slow`, запускается отдельно; лог — в отчёт.

### Р-D. Поправки «спека писана до релизов» (drift отгруженной реальности)
Ратифицировать явно — иначе harness будет ассертить то, чего в отгруженном продукте нет:
1. **Сценарий 4 (lockout).** Спека говорит «pending дропнуты». Отгруженная семантика
   (`APPROVER-PROTOCOL.md` §4, `approver.rs:403-410`): pending-элементы **резолвятся в
   `denied`** (fail-closed) и остаются запрашиваемыми в retention-окне. Ассерт:
   `get_execution_status` → `denied`, а не «исчез».
2. **Сценарий 5 (unlimited-approve).** Спека требует «карточку с UNLIMITED + чекбокс + PIN».
   В отгруженной консоли v0.1.0 **чекбокса нет** (`ui.rs:293-332`): карточка показывает
   `HIGH RISK: unlimited_approval` (bold) + `decoded_call.method: approve` + `spender` +
   `amount: UNLIMITED` (bold) + промпт «High-risk approval — enter your PIN:» с маской.
   **Решение Гейта-1: принять отгруженное как есть** и ассертить эти три вещи (риск, UNLIMITED,
   PIN-гейт): защитная функция чекбокса — «второе осознанное действие» — фактически несёт PIN,
   который строже. Приёмка измеряет продукт, а не чинит его.
   **БЕЗУСЛОВНО (директива Гейта-1):** отклонение от обещания Этапа 3 идёт (а) в отчёт Гейта-2 и
   (б) отдельной задачей в репо `console` — не «если Капитан сочтёт нужным». Иначе обещание
   Этапа 3 тихо испарится, а нам нужно ровно обратное.
3. **Подкоманд `watch`/`approve` у консоли нет** — v0.1.0 это один экран (`main.rs`);
   формулировки сценариев «console watch» читаются как «запуск консоли».

### Р-E. Дом harness'а, запуск и гейты (переписано после /check)
`tests/e2e/` в репо mcp. Образ берётся из env `RUSTOK_E2E_IMAGE`, дефолт
`ghcr.io/rustok-org/rustok-wallet-tui:v0.6.0` — приёмка обязана целиться в конкретный тег.

**Маркера НЕДОСТАТОЧНО, чтобы CI не тронул сюиту** (находка /check): `ci.yml:40` гоняет
`uv run pytest`, `testpaths = ["tests"]` — без явного deselect e2e-тесты соберутся в CI, где нет
ни podman, ни образа. Поэтому в `pyproject.toml`:
```toml
markers = ["e2e: acceptance against the shipped wallet image (needs podman)",
           "e2e_slow: same, but waits out the 15-minute pending TTL"]
addopts = ["-m", "not e2e and not e2e_slow"]
```
Проверено экспериментом: дефолтный прогон деселектит e2e; `pytest -m e2e` в CLI перебивает
`addopts` и гоняет ровно сюиту.

**Гейты harness'а = те, что реально есть в репо:** `ruff check src tests` + `ruff format --check`
(`ci.yml:27-28`) + `pytest`. Mypy в этом репо `src`-скоупный (`ci.yml:29`) — не выдумываем гейт,
которого нет. Bandit-правила `S` включены и на `tests`, поэтому подъём podman'а требует
`per-file-ignores` для `"tests/e2e/*" = ["S603", "S607"]` с обоснованием в конфиге: фиксированный
argv, без `shell=True`, без пользовательского ввода — предусловия правил выполнены (глушилка с
причиной, а не «чтобы позеленело»).

### Р-E2. Изоляция сценариев — свежий кошелёк на каждый (находка /check)
PIN-lockout живёт **5 минут и на диске** (`pin.rs`, data-dir), а не в памяти процесса: S4 (три
неверных PIN) отравил бы S5 (high-risk approve с PIN) ответом `locked` не по вине кода. Поэтому
каждый сценарий получает **свой том + свой свежий кошелёк (`create-wallet`) + свой контейнер с
уникальным именем**; teardown гарантированный (`podman rm -f`, `podman volume rm`) — иначе
упавший прогон блокирует следующий занятым именем. Цена — ~1–2 с на сценарий, приемлемо.

### Р-E3. anvil доступен harness'у напрямую (находка /check)
anvil поднимается в podman-сети **с пробросом порта** (`-p 127.0.0.1:<free>:8545`): кошелёк ходит
к нему по DNS-имени сети (`http://anvil:8545`), harness — с хоста по `127.0.0.1:<free>`. Это
единственный способ (а) налить баланс `anvil_setBalance` и (б) проверить в S1, что tx **реально в
цепи** (`eth_getTransactionByHash`) — с хоста имени `anvil` не существует. Foundry-контейнер ради
`cast` не нужен: RPC — обычный JSON-over-HTTP из Python.

### Р-F. `/run/wallet` под podman (tmpfs-каверза) = сценарий S0, а не «ручная проверка»
Ассерт: сокет `/run/wallet/approve.sock` существует внутри контейнера под podman **и** консоль
к нему подключается (это и есть доказательство, что entrypoint пересоздал каталог на tmpfs).

### Р-G. Upgrade-доки — объём
Секция в `docs/INSTALL.md` (+ ссылка из `TROUBLESHOOTING.md`): апгрейд образа v0.5.x → v0.6.0 —
`podman/docker pull` нового тега, тот же том `rustok-wallet-tui:/data` = тот же кошелёк, ключ и
PIN переживают апгрейд; контейнер пересоздать (`--name` занят); что НЕ переносится — pending-очередь
(она in-memory и умирает с контейнером, §8 родительской спеки). **ТВОЙ ВЫБОР, Капитан:** нужен ли
здесь ещё и путь «переезд с агентской редакции (rustok-wallet) на консольную (rustok-wallet-tui)» —
это разные тома и разные кошельки (по сайту: «new wallet + move funds on-chain»). Рекомендую
**ограничиться апгрейдом внутри консольной линии** (минимум скоупа); межредакционный переезд —
контент сайта, уже сказан там.

## Сценарии (что именно доказываем)

| # | Сценарий | Ключевые ассерты |
|---|---|---|
| S0 | podman-tmpfs | сокет `/run/wallet/approve.sock` есть; консоль подключилась (hello прошёл) |
| 1 | парковка → PIN → `y` → executed | agent: `pending` → `executed`; консоль: `APPROVED`/`executed — 0x…`; **tx_hash идентичен** у консоли и у `get_execution_status`; tx реально в цепи (`eth_getTransactionByHash` в anvil через проброшенный порт, Р-E3) |
| 2 | DENY | консоль: `REJECTED`, exit 4 (`EXIT_REJECTED`); agent: `denied` |
| 3 | истечение (slow) | дедлайн снят **при парковке** (пока `pending`, число живое) и лежит в будущем; после ожидания agent: `state == "expired"` и **`not_after_unix is None`** (Гейт-1); снятый дедлайн к этому моменту — в прошлом; консоль карточку больше не даёт (`unknown_id`) |
| 4 | неверный PIN ×3 | 3-й ответ — `bad_pin, attempts_left: 0`; следующий — `locked`; pending → **`denied`** (Р-D.1) |
| 5 | unlimited-approve | **два слоя гейта.** TUI: карточка показывает `HIGH RISK: unlimited_approval` + `amount: UNLIMITED`; `y` **не отправляет approve**, а открывает PIN-промпт (`app.rs:535-538` — клиентский гейт), верный PIN → executed. Сервер: approve без `pin` на high-risk id (сокет-клиент внутри контейнера) → `pin_required` — через TUI этот ответ недостижим, консоль его не умеет послать (Р-D.2) |
| 6 | `podman exec -i` (pipe) | консоль: «needs an interactive terminal», **exit 3** (`EXIT_NO_TTY`), approve недостижим |
| 7 | APPROVE без AUTH | `podman exec -i <name> python3 -c '<JSON-lines клиент>'` (в образе есть python3+socket — проверено): `hello` → `approve` без `auth` → `unauthorized` |

Сценарий 7 бьёт по протоколу, а не по TUI (консоль сама не даёт послать approve до auth) — это
и есть проверка серверного гейта; клиент — тонкий сокет-клиент, запускаемый **внутри контейнера**
(сокет наружу не смонтирован, с хоста недостижим — находка /check).

**Ни одного `sleep` в качестве синхронизации** (находка /check): каждый шаг ждёт своего
отрендеренного экрана или JSON-ответа с таймаутом — включая экран `APPROVED` после `y`.

## Критерии приёмки

PR готов, когда:
1. `uv run pytest` (дефолт) — зелёный, **156 тестов**, e2e деселектнуты (доказать выводом
   `deselected`), ни один существующий тест не деградировал.
2. `uv run pytest -m e2e` — 7 тестов (S0,1,2,4,5,6,7) зелёные против `v0.6.0`; лог целиком в отчёт.
3. `uv run pytest -m e2e_slow` — S3 зелёный (≈16 мин); лог в отчёт.
4. `uv run ruff check src tests` + `uv run ruff format --check src tests` — зелёные (гейты CI);
   `/python-review` и `/review` — без блокеров.
5. Прод-код mcp не изменён ни на байт (диф = `tests/` + `docs/` + `pyproject.toml`/`uv.lock`/CHANGELOG).
6. Upgrade-секция в доках; версии в примерах = `v0.6.0`.

merged · ветка удалена · гейты зелёные (показанным прогоном) · доки обновлены · PLAN-OF-RECORD
(core) обновлён · workflow-state = idle · находки E2E (если будут) вынесены Капитану отдельным

## Тест-план

Сюита сама и есть тест-план; её собственная корректность держится на двух вещах:
- **Каждый ассерт — на отрендеренном экране или на JSON-ответе**, не на сыром pty-потоке.
- **Негативные сценарии (2,3,4,6,7) обязаны падать красным при мутации:** перед сдачей проверю
  каждый — например, S6 без `-i`-режима (дать tty) обязан покраснеть, S4 с верным PIN обязан
  покраснеть. Тест, который не умеет падать, — не тест (правило №0 `/testing`).

## Доказательства (зонды до Гейта-1, scratchpad, в репо не идут)

Три технических риска сняты **до** написания этой спеки — чтобы спека не утверждала непроверенное:
1. `preview_transaction` против anvil из контейнера кошелька: `estimated_gas: 21000`,
   `revert_check: wont_revert`, `preview_id` выдан; `execute_transaction` → `pending`
   (образ 0.6.0, chain 31337, podman-сеть).
