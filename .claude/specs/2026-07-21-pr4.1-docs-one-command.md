## Цель

Довести документацию до состояния, в котором **первый живой пользователь ставит кошелёк
одной командой**, а не проходит семь болячек руками. Сегодня доки описывают только ручной
путь (`podman run … create-wallet`, ручная правка MCP-конфига, `docker exec` по метке), —
шима `rustok` и инсталлера в них НЕТ вообще, а `INSTALL.md:133` до сих пор обещает их
как «coming». PR переносит центр тяжести на `curl|sh` → `rustok init` → `rustok connect`,
сохраняет ручной путь как честное приложение (для тех, кто не хочет пайп в shell) и
запирает результат греп-инвариантом, как это уже сделано для `--name` и env-file.

Кода продукта PR **не трогает** — только доки + новый тест-гвард.

## Развилки к ратификации (Капитан) — предрешены

### Р-1. Мержим 4.1 ДО того, как релиз существует?

Инсталлер ставится с raw-URL по **immutable-тегу** (ратифицировано, эпик §Гейт-1 раунд-2).
Такого тега ещё нет: `install.sh` живёт только на `main` и до 4.2 несёт fail-closed
плейсхолдеры. Значит, между мержем 4.1 и релизом 4.2 curl-строка в доках на `main` будет
отдавать 404.

- **(а) Рекомендую: мержить 4.1 сейчас**, URL пинить на будущий тег релиза, а ручной путь
  оставить в INSTALL.md разделом ниже — читатель в окне не заперт.
- **(б) Почему:** эпик так и секвенирован (4.1 → 4.2); окно ограничено и почти безлюдно
  (анонса нет, листинги ClawHub/registry всё ещё указывают на 0.7.1); держать
  отревьюенную ветку открытой — ровно то, что сгнило 17.07 (обрыв сессии, код без пуша,
  порванный круг).
- **(в) Контраргумент против себя:** репо публичное, README — витрина; 404 на первой же
  команде — впечатление хуже, чем честное «one-command пока нет».
  **Почему не перевесил:** README ведёт читателя в INSTALL.md, а не в голый curl, и ручной
  путь там же, одним экраном ниже; доки при этом внутренне честны.
- **(г) Цена ошибки:** посетитель скопировал команду → 404 → прочитал ручной раздел →
  поставил руками. Обратимо, деньги/ключи не затронуты.
- **(д) Что переключит выбор:** если ops-шаг (первый подписанный dispatch) откладывается на
  недели, ИЛИ на README `main` уже идёт живой трафик → вариант (б): 4.1 лежит в ветке
  отревьюенной и мержится встык с 4.2.

### Р-2. Какой тег в install-URL

Линия релизов этого продукта в репо — `wallet-tui-v0.5.0 … wallet-tui-v0.7.1`.
Голое пространство `v0.1.0/v0.1.2` занято старым python-пакетом `rustok-mcp`.

- **(а) Рекомендую: продолжить линию — `wallet-tui-v0.8.0`**, URL:
  `https://raw.githubusercontent.com/rustok-org/mcp/wallet-tui-v0.8.0/scripts/install.sh`.
- **(б) Почему:** это тег, который уже возит именно этот продукт; переиспользовать голое
  `vX` — значит положить рядом `v0.1.2` (питон-пакет) и `v0.8.0` (кошелёк) как две
  несвязанные вещи в одном пространстве имён.
- **(в) Контраргумент против себя:** `…/mcp/wallet-tui-v0.8.0/…` длиннее и уродливее, чем
  `…/mcp/v0.8.0/…`, а эпик пишет `vX.Y.Z`. **Почему не перевесил:** URL копипастят, а не
  набирают; двусмысленность истории тегов — постоянная цена, длина — косметика.
- **(г) Цена ошибки:** переименовать git-тег ДО релиза — дёшево; ПОСЛЕ публикации — дорого
  (у пользователей запинены мёртвые URL).
- **(д) Что переключит выбор:** если Капитан намерен ре-базировать версионирование всего
  репо на 0.8.0 (питон-пакет и wallet-tui сходятся в одну версию) → голое `v0.8.0`.

> Номер `0.8.0` — предложение; сам бамп и его фиксация = PR-4.2. Здесь нужен только строковый
> тег, чтобы доки не несли плейсхолдер. Тест запирает: плейсхолдеров нет, все вхождения
> побайтово совпадают.

### Р-3. Куда класть запрет passthrough-пароля (следствие нита-1)

Нит-1 чинит доки, но ничто не помешает passthrough-форме вернуться. Гвард нужен — вопрос,
в какой файл.

- **(а) Рекомендую: ужесточить существующий `test_docs_secret_delivery.py`** — добавить в
  `FORBIDDEN` **два** паттерна passthrough (МИНОР-1 Ревьюера: один их не покрывает):
  1. shell-форма — `-e RUSTOK_KEYRING_PASSWORD \` (хвостовой ` \` отсекает `_FILE=`-строки);
  2. JSON-форма (args MCP-конфига) — `"RUSTOK_KEYRING_PASSWORD"` в кавычках с обеих сторон.

  **Оба дискриминируют, проверено грепом по 5 докам, а не рассуждением:** JSON-паттерн даёт
  сегодня **0 вхождений** (значит, не красит текущие доки), и не совпадает ни с одной из
  7 строк `target=RUSTOK_KEYRING_PASSWORD`, ни с одной из 10 строк `…_PASSWORD_FILE`
  (закрывающая кавычка после `D` отсекает и то, и другое). Форма не гипотетическая:
  `SKILL.md:150` уже несёт ровно такой passthrough на RPC — `"-e", "RUSTOK_RPC_URLS_1"` с
  `env`-блоком; перенести его на пароль — движение в один символ.
  **Честная слабость паттерна:** он сработает и на прозе вида `the "RUSTOK_KEYRING_PASSWORD"
  variable`. Риск принят: доки последовательно пишут идентификаторы в бэктиках (сверено), а
  ложный RED виден на локальном прогоне до пуша.
- **(б) Почему:** докстринг этого файла уже объявляет ровно этот инвариант («never an inline
  password value, an env-block value, or the legacy `--env-file` recipe»); passthrough — дыра
  в его собственном покрытии, а не отдельная тема. Класть его в новый one-command-гвард —
  значит размазать один инвариант по двум файлам, и следующий читатель не найдёт его там,
  где ищет.
- **(в) Контраргумент против себя:** моя же спека объявила «существующие гварды не трогаем», и
  правка чужого гварда в докс-PR — это расширение блast-радиуса.
  **Почему не перевесил:** это ужесточение (+1 запрет), а не ослабление; риск — только
  ложный RED на самом PR, который я обязан прогнать локально до пуша в любом случае.
- **(г) Цена ошибки:** если паттерн окажется слишком широким — краснеет свой же гвард на
  локальном прогоне, до пуша. Продукта не касается вообще.
- **(д) Что переключит выбор:** если ты держишь правило «докс-PR не трогает чужие тесты» как
  жёсткое — кладу тот же запрет в новый `test_docs_one_command.py` (функционально
  эквивалентно, хуже только по месту жительства инварианта).

## Критерии приёмки — «PR готов, когда…»

1. **Один путь читается как основной.** В INSTALL.md и README первым идёт `curl|sh` →
   `rustok init` → `rustok connect`; ручные рецепты явно помечены как приложение
   «если не хотите пайп в shell / ставите без шима».
2. **Ни одной несуществующей команды.** Каждая команда/флаг в доках существует в
   `cli/rustok` (сверено с `usage()` и диспатчем) и в `scripts/install.sh`
   (`RUSTOK_NO_MODIFY_PATH`, PATH-блок, откат re-run по старому тегу).
3. **inspect-before-run задокументирован** — форма «скачать → прочитать → выполнить», не
   только пайп.
4. **cosign назван предусловием** с ссылкой на установку: без него инсталлер умирает
   именованной ошибкой, а «одна команда» без этого — ложь.
5. **Честность про update — при КАЖДОМ упоминании** (нит-2 Ревьюера). Везде, где
   документируется обновление, стоит явная оговорка: `rustok update` тянет образ **по тегу и
   без cosign-проверки** — гарантия подписи даётся на момент установки, не на весь жизненный
   цикл (долг verify-on-update открыт). Ни одна формулировка не читается как «supply-chain
   закрыт целиком». Сегодня таких мест два (сверено грепом): `INSTALL.md:156` «Upgrading the
   wallet image» и `TROUBLESHOOTING.md:79` «After an upgrade…».
   **Форма ассерта — условная по всем `DOC_PATHS`** (МИНОР-2 Ревьюера: поимённый список из
   двух файлов не уронится ни от третьего файла, ни от нового упоминания внутри уже
   покрытого): *файл упоминает `rustok update` → файл несёт маркерную оговорку*. Два
   поимённых файла остаются **позитивными контролями** — доказывают, что условие вообще
   срабатывает, а не что «ни один файл не упомянул update». Такой гвард переживает 4.2 и
   Стадию 5 без правок.

5-бис. **Ручной аппендикс несёт ИСПРАВЛЕННЫЙ рецепт** (нит-1 Ревьюера). Сегодня
   `SKILL.md:60,64` отдаёт пароль passthrough'ом (`export RUSTOK_KEYRING_PASSWORD` + голый
   `-e RUSTOK_KEYRING_PASSWORD`): значение не в argv, но видно в `inspect` `Config.Env` — тот
   же класс утечки, что и ретайрнутый `--env-file`, только тише, и текущие 5 FORBIDDEN-
   паттернов его не ловят (им нужен `="` / `": "` / `=%s`). В новом ручном аппендиксе
   онбординг идёт **только** через `podman secret` / `_FILE`; passthrough-формы не остаётся
   ни в одном доке.
6. **Честность про ключи.** `uninstall` описан как data-safe; `--purge-keys` — с двойным
   гейтом и прямым предупреждением о невосстановимости без сид-фразы.
7. **Ноль плейсхолдеров** в исполняемых командах; тег в install-URL одинаков во всех файлах.
8. **Оба существующих гварда зелёные** (`test_docs_container_discovery.py`,
   `test_docs_secret_delivery.py`) — их позитивные контроли (`podman secret create` и
   `RUSTOK_KEYRING_PASSWORD_FILE` в INSTALL+SKILL, `--filter label=rustok.agent=` в INSTALL)
   продолжают выполняться за счёт сохранённого ручного приложения.
   **Единственная правка гварда — Р-3 ниже (ужесточение), под ратификацию Капитана.**
   Ослабление гварда не предлагается ни в каком виде.
9. **Стейл-обещание снято** (`INSTALL.md:133` «A short `rustok` command … is coming»).

## Тест-план

**Новый гвард `tests/test_docs_one_command.py`** — в стиле двух существующих
(FORBIDDEN-паттерны + позитивные контроли, чтобы «доки замолчали» не проходило):

*Позитивные контроли (доки реально учат one-command):*
- INSTALL.md и README.md содержат curl-строку с `--proto '=https' --tlsv1.2` и путём
  `/scripts/install.sh`;
- INSTALL.md содержит `rustok init`, `rustok connect`, `rustok console`, `rustok doctor`,
  `rustok uninstall`;
- INSTALL.md документирует `cosign` как предусловие и `RUSTOK_NO_MODIFY_PATH`;
- INSTALL.md несёт inspect-before-run (скачивание в файл до запуска);
- SKILL.md ведёт пользователя через `rustok init` / `rustok connect claude`.

*Запреты:*
- ни в одном доке нет плейсхолдера версии в исполняемой команде (`vX.Y.Z`, `<version>`,
  `v0.0.0`);
- снято обещание «is coming» про обёртку discovery;
- **инвариант согласованности:** все вхождения install-URL во всех доках побайтово равны
  (страховка от «4.2 бампнул один файл из трёх»).

*Честность (нит-2 + МИНОР-2 — условный ассерт, не поимённый список):*
- по **всем** `DOC_PATHS`: файл, упоминающий `rustok update`, обязан нести маркерную
  оговорку про тег/отсутствие cosign;
- позитивные контроли: `INSTALL.md` и `TROUBLESHOOTING.md` действительно упоминают
  `rustok update` (иначе условный ассерт зелёный вхолостую).

*Passthrough-пароль (нит-1 + МИНОР-1 — ДВА паттерна, место по итогу Р-3):*
- ни один док не содержит shell-формы `-e RUSTOK_KEYRING_PASSWORD \`;
- ни один док не содержит JSON-формы `"RUSTOK_KEYRING_PASSWORD"` в args MCP-конфига.

**Мутационные пруфы (обязательны, каждый гвард падает своим тестом).** Для каждого класса
ассертов — точечная мутация дока, прогон, RED, откат:
1. убрать curl-строку из README → падает позитивный контроль;
2. вернуть `vX.Y.Z` в команду → падает запрет плейсхолдера;
3. разойтись тегом в README vs INSTALL → падает инвариант согласованности;
4. убрать оговорку про update → падает честностный ассерт;
5. вернуть строку «is coming» → падает запрет;
6. **(нит-2)** убрать оговорку ТОЛЬКО из `TROUBLESHOOTING.md`, оставив в `INSTALL.md` →
   гвард обязан покраснеть (ассерт пофайловый, а не «хотя бы где-то»);
7. **(МИНОР-2 — то, что поимённый список НЕ ловит)** дописать упоминание `rustok update`
   в **третий** файл (README) без оговорки → гвард обязан покраснеть. Это и есть пруф, что
   ассерт условный по всем `DOC_PATHS` и переживёт 4.2/Стадию 5;
8. **(нит-1, shell)** вернуть `-e RUSTOK_KEYRING_PASSWORD \` в ручной блок SKILL.md →
   падает shell-запрет. Контроль базы: тот же паттерн прогнать против **текущего**
   `SKILL.md:64` — обязан ловиться (иначе запрет написан мимо реальной формы);
9. **(МИНОР-1, JSON)** вписать `"-e", "RUSTOK_KEYRING_PASSWORD",` в args MCP-конфига
   SKILL.md (форма-донор уже стоит рядом на RPC, `SKILL.md:150`) → падает JSON-запрет.
   Без этой мутации второй паттерн остался бы недоказанным — ровно тот «гвард уже, чем

## Замечено, не трогаю

- **cosign — жёсткое предусловие «одной команды».** `install.sh:106` умирает без него.
  Для массового пользователя «поставь cosign, потом одна команда» — заметное трение.
  Ослаблять нельзя (это и есть verify-first-гарантия). Кандидат в бэклог: вшить проверку
  подписи иначе (bundle + встроенный верификатор) — отдельным решением, не здесь.
- Память несёт заметку «mcp/main SKILL.md стейл 0.3.2» — **устарела**: в файле `version: 0.7.1`.
  Заметку поправлю при обновлении памяти, кода не касается.
- `AGENTS.md` §Release Checklist не знает про install.sh/шим — материал для 4.2, не для 4.1.
- `rustok update` без cosign (verify-on-update) — код, отдельный PR; здесь только честная
  формулировка.
