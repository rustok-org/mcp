# Мини-круг: переговоры версии MCP-протокола + serverInfo из метаданных → v0.7.1

> Статус: **ПРЕДЛОЖЕНИЕ, ждёт Гейта-1.** Кода нет до двух подписей.
> Триггер: первый живой пользователь (Капитан) — Claude Code 2.1.2 просит
> `protocolVersion: "2025-11-25"`, сервер отвечает прибитым `"2024-11-05"`
> (handlers.py:96), клиент ревизию не принимает → молчание → таймаут 30с →
> «Failed to connect». **Флагманский листинг не подключается ни у одного
> пользователя текущего Claude Code.** Диагностика задокументирована в
> console/workflow-state (прослушка tee: оба потока живы, осталась версия).

## Цель

`initialize` отвечает **ревизией клиента**, если она из поддержанного набора
(для tools-only сервера ревизии 2024-11-05…2025-11-25 совместимы по нашей
поверхности), иначе — нашей новейшей (по спеке MCP клиент тогда сам решает).
`serverInfo.version` читается из метаданных пакета — хардкод «0.6.0» в коде
(второй гвоздь, из-за него даже пересобранный образ врал бы) уходит навсегда.

## Скоуп

1. **handlers.py**: `SUPPORTED_PROTOCOL_VERSIONS = ("2025-11-25", "2025-06-18",
   "2025-03-26", "2024-11-05")`; негошиация: клиентская строка из набора → echo;
   незнакомая/отсутствует/не строка → первая (новейшая). `serverInfo.version` =
   `importlib.metadata.version("rustok-mcp")`.
2. **Тесты** (red→green): echo каждой из 4 ревизий · незнакомая → новейшая ·
   отсутствие поля → новейшая (defensive) · не-строка → новейшая ·
   serverInfo.version == metadata. Существующие 3 ассерта «== 2024-11-05»
   (test_stdio:28, test_handlers:21, test_sse:140): их фикстуры protocolVersion
   НЕ шлют (проверено /check) → по новой логике получат новейшую — все три
   ассерта ОБНОВЛЯЮТСЯ на неё (единственные правки существующих тестов,
   поимённо). `params` бывает `None | list | dict` (protocol.py:25, /check) —
   негошиация читает версию только из dict, всё прочее = «не прислали».
3. **Живая приёмка самого бага**: `claude mcp list` на этой машине →
   `rustok … ✓ connected` (то, что сломано сейчас), затем реальная агентская
   сессия Капитана видит инструменты.
4. **Релиз v0.7.1**: манифесты 0.7.0→0.7.1 (pyproject+lock, SKILL.md
   frontmatter+3 тега+health-check, server.json, claw.json — анти-дрейф-тест
   держит) · Dockerfile.wallet — только примеры в шапке (пины core/console НЕ
   меняются) · INSTALL/TROUBLESHOOTING · CHANGELOG [0.7.1] · сборка+пуш
   v0.7.1+v0.7+v0 (по «добро») · clawhub publish 0.7.1.

**НЕ входит:** новые capabilities/фичи ревизий (elicitation/roots — клиентские),
SSE-транспорт сверх общего handler-пути, e2e-переписывание (mcp_client всеяден —
это отдельный бэклог-вопрос честности приёмки: она баг не поймала).

## Критерии приёмки

1. Все 4 ревизии echo'ются (тест); незнакомая/отсутствует/мусор → новейшая.
2. `serverInfo.version` из метаданных (тест), хардкода в src нет (греп).
3. Гейты mcp дословно зелёные (ruff/format/mypy/pytest; shellcheck — CI).
4. **`claude mcp list` → connected** на машине Капитана (живой баг закрыт).
5. Образ v0.7.1 в GHCR (все 3 тега), ClawHub latest=0.7.1, initialize против
   образа возвращает клиентскую ревизию и version 0.7.1 (probe-прогон).

## Решения (предрешены)

1. **Echo клиентской, не всегда-новейшая** — (а) так делает эталонный SDK;
   (б) незнакомому клиенту наша новейшая по спеке; (в) контраргумент «всегда
   отвечать новейшей» — не перевесил: старый клиент (наш же e2e) остался бы
   жив только случайно; (г) цена: ничего; (д) переключатель: слово Ревьюера.
2. **Патч-версия 0.7.1** (не 0.8.0) — фикс совместимости, поверхность
   инструментов не менялась.
3. **e2e не трогаем в этом круге** — их всеядность к версии = отдельный
   бэклог-пункт (приёмка должна была поймать этот баг и не поймала).

2. INSTALL-уроки первого пользователя (-d -i, кавычки, --entrypoint) — отдельный
